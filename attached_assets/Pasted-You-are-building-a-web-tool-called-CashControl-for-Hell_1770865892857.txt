You are building a web tool called CashControl for Hello Sugar to track and reconcile physical cash against Boulevard.

Goal
Replace our current Google Form + Sheet process with a system that:
- makes cash counting easy for estheticians
- provides early detection and real-time alerts when something is off
- logs receipts for any cash spent
- supports audits, reporting, and periodic cash collections

Core concepts
- We operate in multiple markets. Each market has multiple locations.
- Each location can be either:
  1) Suite location: up to 2 suites, each suite counts cash separately
  2) Flagship location: one pooled cash container for the whole location, regardless of rooms
- “Cash container” means the envelope/till that is being counted (suite container or flagship container).

Source of truth
- Boulevard is the source of truth for “cash expected.”
- The tool must support two ingestion methods for Boulevard cash transactions:
  1) Direct Boulevard API integration (preferred, if credentials are available)
  2) CSV upload fallback (required)
- We may start with CSV and later switch to API, so the design must support both without changing the rest of the system.

Boulevard data fields we need to store for each cash transaction
- date/time
- location
- appointment ID
- amount
- staff name
- client name
- (payment type if available; the system should be able to filter/import only cash transactions)

Cash expected math
Use this definition:
cash_should_be_there = beginning_cash + cash_collected_from_cash_appointments - cash_spent_with_receipts

Where:
- beginning_cash is what was confirmed at start of shift (based on prior shift end)
- cash_collected_from_cash_appointments comes from Boulevard transactions during the shift
- cash_spent_with_receipts is any receipts recorded for that shift/container

Flagship special rules
We are changing flagship handling:
- Each day starts with $20 in the till for change
- At end of day, someone counts cash and drops it into a safe slot
- Nobody but the owner has access to the safe
- Therefore flagship cash effectively resets daily (does not accumulate)
- The tool must support this daily reset logic and the $20 daily starting float

Shift workflow (esthetician-facing)
- Shifts are defined by clocking in/out, but we are not integrating with timeclock in v1.
- The tool should assume “start shift” happens at submission time and “end shift” happens at submission time.
- Estheticians must submit BOTH:
  1) Start-of-shift count
  2) End-of-shift count

Start-of-shift submission requirements
- Esthetician selects:
  - their name from a dropdown (admin-managed list)
  - location
  - suite container (if suite location; flagship is just the location container)
- Tool shows:
  - prior shift end amount for that same container
  - expected amount according to Boulevard (and receipts) for that moment
- Esthetician enters the counted cash amount
- If the counted amount does not match expected:
  - they must enter a reason note before they can submit
  - management must be alerted immediately with details

End-of-shift submission requirements
- Tool shows:
  - the start-of-shift amount for that shift
  - how much cash should have been added based on Boulevard during that shift
  - expected end amount
- Esthetician enters the counted end amount
- If mismatch:
  - require a reason note before submit
  - management must be alerted immediately with details
- If end amount is less than start amount:
  - warn the esthetician and ask if cash was spent
  - if yes, they must upload receipt(s)
  - if no, require a note explaining why it decreased

Receipt requirements
- Any time cash is spent, a receipt photo/file must be uploaded.
- Support uploads: HEIC, JPG, PNG, PDF.
- The tool should attempt to extract the receipt amount automatically (OCR), then show the extracted amount for confirmation.
- The user must be able to correct the amount if OCR is wrong.
- No receipt categories or approval rules are needed; we just need documentation.
- Management must be alerted every time a receipt is submitted (always).

Alerts and early detection (required)
- Alert types:
  1) Start-of-shift mismatch
  2) End-of-shift mismatch
  3) End-of-shift missing when start-of-shift exists
  4) Receipt missing when cash appears to have decreased / cash spent is indicated
  5) Receipt submitted (always notify)
  6) Cash collection mismatch (see below)
- Tolerance: any mismatch triggers (no thresholds).
- Missing end-of-shift detection:
  - if a start-of-shift submission exists but no end-of-shift submission occurred, create an alert at 9:00pm each night (local time per location if possible).
- Alerts should be real-time whenever possible.

SMS notifications via Quo (required)
- Integrate with Quo SMS (we have an account and can provide an API key).
- Allow multiple phone numbers to receive alerts (admin configurable).
- SMS messages must include:
  - alert type
  - staff name
  - market + location + container
  - expected vs actual values (when relevant)
  - timestamp
  - discrepancy note (when relevant)
  - a link to open the record in the admin portal (if possible)

Roles and access
- Estheticians:
  - no login
  - can only access the counting workflow + receipt upload
  - can see the prior shift count for the same container only (not full history)
- Managers and Owner:
  - admin portal access
  - can view reports, variances, receipts, and alerts
  - can configure markets/locations/suites/esthetician dropdown list
  - can configure alert phone numbers
- Admin authentication:
  - Google login required
  - only hellosugar.salon email addresses allowed
  - AND only allowlisted users can access (not every email in the domain)

Manager/Owner admin features (requirements)
- Setup:
  - manage markets
  - manage locations (suite vs flagship, add suites/containers, timezone if needed)
  - manage esthetician name list (dropdown)
  - manage admin allowlist
  - manage alert recipient phone numbers
- Boulevard data ingestion:
  - CSV import (required)
  - Direct API integration (optional but included in v1 if possible)
  - ability to manually input “Boulevard cash expected totals” for a day/location as a backup workflow (if transaction-level import is not possible)
- Dashboards / reports:
  - open variances today
  - missing end-of-shift submissions
  - cash position by location/container
  - receipts submitted
  - receipt missing situations
  - variance trends over time
  - filters: market, location, container, staff, date range
- Exports:
  - exportable audit logs for any date range (CSV)
- Alerts log:
  - view all alerts, status, details, linked shift/receipt, and timestamps

Cash collection events (requirements)
- Every couple months we physically collect cash.
- The tool must support a “cash collection event” workflow:
  - select location/container(s)
  - tool calculates how much cash should be present since last collection event (based on Boulevard and receipts)
  - collector enters how much cash was actually collected
  - if mismatch, require a note and alert management
  - after collection, suite containers reset to $0
  - (flagship follows its daily reset logic, but collection should still be auditable)

Data retention
- Keep records forever by default
- Provide archive/delete controls for admins if needed

Build expectations
- Mobile-friendly UI (esthetician workflow must be fast)
- Strong auditability: every submission/receipt/collection should be traceable
- Avoid human error by showing prior shift numbers and expected amounts clearly
- The system should handle that Boulevard data may exist before estheticians start using the tool

Deliverables
- Build the tool with these requirements
- Provide setup instructions
- Include a way to configure Quo API key and admin allowlist
